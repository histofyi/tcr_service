# TCR service — data layer (mock for S3 prebake)

Prebaked static data for the histo.fyi **TCRs** section. In production these
files live in S3 and are served directly; DuckDB reads the two `*.parquet`
indexes for the sortable `/tcrs` and `/clonotypes` browse. This folder mocks
that layout inside the web-app repo (`data/`).

Generated from master table v17 (class_i_annotation.csv, 206 canonical
structures) + com_coords.json + pipeline tables (contacts, BSA, SC,
completeness) + literature_summaries.json + pdbe_publications.json +
PDBe deposition/release dates. Parquet engine: **fastparquet**.

## Page units
- **TCR** (123) — the primary page unit (`/tcrs`, `/tcrs/{tcr_id}`).
  `tcr_id` = slugified TCR name; the 2 unnamed → `hghv4` (9D95),
  `ki-890` (9K2R — donor code; no published clone name, Chikata et al. Nat Commun 2025). Affinity-engineered variants
  (name suffix `*`) are grouped into the parent TCR page and flagged
  `has_engineered_variants`; the residue-level engineering detail lives on the
  clonotype.
- **clonotype** (133) — finer grain (`/clonotypes`, `/clonotypes/{clonotype_id}`).
  Unique (TRAV,TRBV,CDR3α,CDR3β). A named TCR may span several clonotypes
  (engineered families e.g. 1G4, TCR156).
- **structure** (206) — the deep-dive leaf (`/tcrs/{tcr_id}/structures/{pdb_id}`).

## Files
- `tcrs.parquet` / `tcrs_index.json` — TCR index (sort keys: tcr_name,
  n_structures, first_deposition_date). List columns are JSON-encoded in parquet.
- `clonotypes.parquet` / `clonotypes_index.json` — clonotype index.
- `tcr/{tcr_id}.json` — per-TCR: header, genes, publications (+exec summaries),
  structure list (with per-structure com_px + engineered flag), variability
  (per-structure footprint-centre spread across ASU copies/altlocs, Å), alleles/
  peptides/antigen_types.
- `clonotype/{clonotype_id}.json` — per-clonotype: genes, CDR3 identity,
  structures, engineered_variants[] (pdb, CDR3 comparison to baseline, paper,
  patent placeholder).
- `structure/{pdb_id}.json` — per-structure deep-dive: resolution/method/dates,
  allele/peptide, CDRs, publication + exec summary, non-canonical flag,
  com_px (6 COM layers, viewer pixel space), contacts_loop_region (chord source:
  CDR-loop × MHC-region contact counts), bsa[18], shape_complementarity[18],
  completeness (per copy).

## Known limitations / TODO
- 9K2R has NO published functional clone name (Chikata et al., Nat Commun 2025) — it is
  identified only by donor code KI-890 + TRAV24*01/TRBV27*01. Slug resolved to `ki-890`
  (was `unknown_1`); bundle carries `donor_code` + `naming_note`.
- Engineered-variant residue detail currently compares CDR3 only; affinity
  maturation is often in framework positions. Residue-level mutation lists +
  patents need per-paper curation (future lit work). Paper links are in place.
- `contacts_loop_region` is loop×region contact COUNTS (from cdr_region_contacts).
  The interactive chord diagram will additionally need residue-residue pairs
  (grant contact_maps format) — a later pipeline pass over the structures.

## AI provenance (REQUIRED on render)
Every `executive_summary` was written by a literature research agent, not a human. Each bundle carries `executive_summary_attribution` (plain) and `executive_summary_attribution_html` = `<smaller>Summary generated by a literature research agent in Claude Science</smaller>`. The front end MUST render this disclosure with the summary, in the `<smaller>` style, everywhere a summary appears (per-structure deep dive AND per-TCR page). Do not display a summary without it.
